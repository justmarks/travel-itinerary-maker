## Summary

**Total: 9 bugs filed.** Severity breakdown:

- **Blocker (1):** Microsoft sign-in cannot complete — sandbox network policy blocks the consumer Microsoft login CDN (`logincdn.msauth.net`, `logincdn.msftauth.net`, `lgincdnmsftuswe2.azureedge.net`). This is an environment/sandbox configuration issue, NOT an app bug, but it gates this entire QA session. All post-auth scenarios (trip CRUD, segments, todos, email scan, sharing, calendar sync, calendar TZ correctness) were therefore unable to be exercised live.
- **High (5):** `/trips` unauthenticated shows a destructive red error instead of routing to login; `/shared/<unknown>` copy + CTA mismatch between desktop and mobile (re-confirmed from previous session); mobile parity gap on `/m/welcome`, `/m/privacy`, `/m/terms`, `/m/release-notes` (all 404); bare Next.js 404 on `/settings`, `/m/settings`, and any `*/fake-id` route instead of an in-app branded 404; desktop login provider buttons have no visible keyboard focus indicator.
- **Medium (2):** Privacy Policy opening sentence still says "Gmail" only when the app supports Outlook too; inconsistent demo-mode link label ("Try with demo data" vs "Try the demo").
- **Low (1):** Login page tab order — first focusable element is the wordmark, not the primary CTA.

**What worked smoothly:**
- All public marketing pages (`/`, `/welcome`, `/privacy`, `/terms`, `/release-notes`) render correctly on both desktop viewport and 375×812 mobile UA.
- `/auth/callback` error handling (4 variants tested: `?error=access_denied`, `?error=server_error`, `#error=invalid_request`, `?code=invalid_code_12345`) all render branded error states with `Back to sign-in` CTA — no React hydration error #418 reproduced. Either it's been fixed since the previous session's notes or it was transient.
- Dark mode renders cleanly on `/welcome`, `/login`, `/release-notes`, `/privacy`, `/m/login` when `prefers-color-scheme: dark` is set.
- Mobile `/m/trip` and `/m/settings/account` correctly redirect to `/m/login` when unauthenticated.
- Desktop `/settings/account` correctly redirects to `/login` when unauthenticated.
- Welcome marketing CTAs route to `/login` as expected. Footer links to release notes / privacy / terms / `mailto:support@itinly.app` all resolve.
- Google sign-in path opens the real Google consent screen (could not proceed without Google test credentials, but the redirect/return mechanics look healthy through that step).

**What was NOT covered and why:**
- Trip CRUD (create, rename, redate, delete), rapid optimistic-update tests
- Email scan: connect Outlook, run scan, dedup, review queue accept/edit/reject
- Segments: add/edit/delete for every type, overnight flights, multi-day hotels, missing-field edge cases
- Todos: per-category add, check/uncheck rapidly, reorder, delete
- Sharing: create link, open in incognito, verify read-only view, revoke, verify revoked 404
- Calendar sync: connect Outlook Calendar, push trip, verify wall-clock TZ correctness per city
- Desktop ↔ mobile parity inside `/trips/[id]` and `/m/trip` — primary actions
- Console / network errors on authenticated routes

**Root cause:** the test account `itinlytest@outlook.com` is a consumer Microsoft account. The OAuth flow lands on `login.live.com`, which loads its login form bundle from `logincdn.msauth.net`, `logincdn.msftauth.net`, and `lgincdnmsftuswe2.azureedge.net`. All three return `403 host_not_allowed` from the sandbox proxy. The MS login page therefore renders blank — the email field appears (served from `login.microsoftonline.com` which IS allowed), but after Next the password form never paints (see `docs/qa-screenshots/ms-after-email.png`). See **[BLOCKER]** below for the exact hostnames the network policy needs.

I attempted Google sign-in as a fallback. `accounts.google.com` is reachable and the Sign-in form rendered — but no Google credentials were provided for the throwaway account, so I could not complete that flow either. The task only supplied MS creds.

I did **not** fall back to `?demo=true` — the mock client doesn't persist, and the task explicitly excludes it.

## Bugs

### [SEV: blocker] Microsoft sign-in unreachable from sandbox — login CDN hosts denied
**Where:** `/login` desktop, `/m/login` mobile → redirects to `login.live.com` after email step
**Repro:**
1. Visit `https://preview.itinly.app/login`.
2. Click "Sign in with Microsoft".
3. Page redirects through `https://login.microsoftonline.com/common/oauth2/v2.0/authorize?...` (loads OK).
4. Enter `itinlytest@outlook.com`, click Next.
5. Page redirects to `https://login.live.com/oauth20_authorize.srf?...` (consumer Microsoft Account flow).
6. Page renders blank/white — no password field.

**Expected:** Microsoft consumer password form renders, accepts `Squiggl3`, completes OAuth, returns to `/auth/callback`, and lands authenticated user on `/trips`.

**Actual:** Page is blank. Browser console shows many `host_not_allowed` 403s plus downstream `CORS policy` errors from origin `https://login.live.com`.

**Notes:** Sandbox proxy denies these hosts (verified with `curl`):
- `logincdn.msauth.net` → `403 x-deny-reason: host_not_allowed`
- `logincdn.msftauth.net` → `403 host_not_allowed`
- `lgincdnmsftuswe2.azureedge.net` → `403 host_not_allowed`

These are the script + brs-config + fluent-UI bundles for the consumer login form. Allowed: `login.microsoftonline.com`, `login.live.com` (the HTML loads), `graph.microsoft.com`, `*.supabase.co` (verified `acpcnnchrgoepygukxxs.supabase.co` 404s gracefully). The enterprise `aadcdn.msauth.net` / `aadcdn.msftauth.net` ARE allowed, but the *consumer* CDN is not.

Per the task's pre-flight instructions ("If any return `host_not_allowed`, STOP and report which hosts the network policy still needs to allow") this is the report. **All post-auth coverage is blocked until those three hosts are allowlisted, or until a Google test account with creds is provided.**

Screenshots: `docs/qa-screenshots/ms-after-email.png` (blank page), `docs/qa-screenshots/ms-loginstep.png` (email step renders).

### [SEV: high] Mobile parity gap — `/m/welcome`, `/m/privacy`, `/m/terms`, `/m/release-notes` all 404
**Where:** `/m/welcome`, `/m/privacy`, `/m/terms`, `/m/release-notes` (mobile shell)
**Repro:**
1. On mobile UA (375×812): visit `https://preview.itinly.app/m/welcome`.
2. Repeat for `/m/privacy`, `/m/terms`, `/m/release-notes`.

**Expected:** Each route either renders an `/m`-shelled version of the corresponding desktop page, OR redirects to the desktop equivalent that already happens to be responsive. Either is fine — a bare 404 is not, because:
- The pages are linked from outside the app (privacy policy URL surfaced to OAuth consent screens, release-notes shared via blog/social, etc.).
- A user who installed the PWA and is browsing `/m/...` will end up on `/m/privacy` if they tap "Privacy" from the in-app menu (assuming it exists post-auth) and expect to land on a real page.

**Actual:** All four return HTTP 404 with the bare Next.js default "404 / This page could not be found." page — no app chrome, no logo, no nav back.

**Notes:** Desktop `/welcome`, `/privacy`, `/terms`, `/release-notes` all work fine (verified — beautiful, on-brand). The `/m` shell exists at `apps/web/src/app/m/` per the source tree, but no `page.tsx` is present under those directories — only `login`, `settings/account`, `shared/[token]`, `trip`. The mobile shell is incomplete. ### [SEV: high] Desktop login provider buttons have no visible keyboard focus indicator
**Where:** `/login` desktop ("Sign in with Google", "Sign in with Microsoft")
**Repro:**
1. Visit `https://preview.itinly.app/login`.
2. Press Tab once → focus lands on the itinly wordmark link (visible 4px cyan ring — looks good).
3. Press Tab again → focus moves to "Sign in with Google" button.
4. Press Tab again → focus moves to "Sign in with Microsoft" button.

**Expected:** Both provider buttons show a visible focus ring (e.g. the same 4px cyan `--primary` ring the wordmark link uses). Keyboard users must be able to tell what's focused before pressing Enter.

**Actual:** Computed style on each focused button shows `outline: ... none ...` (suppressed default) and `box-shadow: ... transparent ... transparent ...` (custom rings are present but all transparent / 0px). Visually, nothing changes when the button receives focus — see `docs/qa-screenshots/polish-login-tab2.png` (Google focused) and `polish-login-tab3.png` (Microsoft focused). Both look identical to the unfocused state.

**Notes:** Verified by reading `document.activeElement` in DevTools — the buttons do receive focus, the indicator just isn't drawn. Mobile `/m/login` uses default browser auto-outlines (`rgb(229, 151, 0) auto 1px`) so focus IS visible there — this is a desktop-only regression. CLAUDE.md calls out WCAG 2.1 AA contrast requirements; focus visibility is part of the same accessibility commitment.

Screenshots: `docs/qa-screenshots/polish-login-tab1.png` (wordmark, ring visible), `polish-login-tab2.png` (Google, no ring), `polish-login-tab3.png` (Microsoft, no ring), `bug-focus-google-no-ring.png`, `bug-focus-google-marked.png` (same shot with a red outline injected to prove which button was focused).

- **Microsoft `aadcdn.msauth.net` / `aadcdn.msftauth.net`** — these ARE allowed, so an **enterprise** Microsoft tenant sign-in might complete in this sandbox. Only the consumer (`login.live.com` + `logincdn.*`) flow is blocked.
