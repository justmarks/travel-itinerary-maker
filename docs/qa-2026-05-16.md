# QA Report — Staging Shakedown (2026-05-16)

**Tester:** Claude (Opus 4.7) running in Claude Code remote-execution sandbox
**Target:** https://preview.itinly.app (Vercel SSO bypassed via share link, Railway backend `itinly-preview.up.railway.app`)
**Branch:** `claude/qa-test-travel-app-mmGim`

> **Note:** This pass got hard-blocked on Microsoft sign-in (sandbox network policy denies `aadcdn.msftauth.net` / `aadcdn.msauth.net`, so the MS login UI renders blank — no inputs in the DOM). All findings below are from unauthenticated routes. A follow-up session with the MS CDNs allowlisted is needed to cover the post-login surface (trips, segments, todos, email scan, sharing, calendar sync, optimistic-update wiring, desktop↔mobile parity inside `/trips/*` and `/m/trip/*`).

### What worked smoothly

- All unauthenticated routes (`/welcome`, `/login`, `/m/login`, `/privacy`, `/terms`, `/release-notes`) render correctly on both desktop and mobile, return 200, and complete first paint in under 3 seconds.
- Login redirect flow is wired: hitting `/`, `/m`, or `/m/login` while logged-out lands on the correct sign-in page; mobile UA routes through `/m/login`, desktop UA through `/login`.
- The `/auth/callback` page correctly distinguishes the two error variants in copy:
  - No `error` param → "Sign-in could not be completed. The provider redirect did not return a session."
  - `error=access_denied` → "Sign-in was cancelled."
- The unknown-share-link state on `/shared/<id>` renders a friendly fallback rather than a hard error.
- Page titles are set per-route (e.g. `Privacy Policy — itinly`, `Release notes — itinly`) — no orphan default `itinly` titles on content pages (except `/login`, which is fine).

### What I could NOT cover, and why

**Microsoft sign-in is unreachable from this sandbox.** Clicking "Sign in with Microsoft" on `/login` redirects to `login.microsoftonline.com/common/oauth2/v2.0/authorize` (which the sandbox allows), but the Microsoft login UI itself is rendered by JS+CSS bundles served from `aadcdn.msftauth.net` and `aadcdn.msauth.net`. Both return `403 host_not_allowed` from this sandbox:

```
$ curl -sI https://aadcdn.msftauth.net/ests/2.1/content/cdnbundles/converged.v2.login.min.css
HTTP/2 403
x-deny-reason: host_not_allowed
```

So the Microsoft sign-in page renders **completely blank** — no inputs, no forms, no body text (verified via `document.querySelectorAll('input').length === 0` and `document.body.innerText === ''`). I can't fill the email field because there isn't one in the DOM. See `docs/qa-screenshots/03b-ms-state.png`. The sandbox network policy needs `aadcdn.msftauth.net`, `aadcdn.msauth.net`, and the live.com host family (the test mailbox is a personal MSA at `@outlook.com`) added to its connect-src allowlist before any future session can run a real end-to-end auth pass.

Google sign-in might be reachable (`accounts.google.com` returns 302 from the sandbox) but I was only given Microsoft credentials.

**Net effect:** every flow that requires an authenticated session is untested — `/trips/*`, `/settings/*`, email scanning, calendar sync, share-link *creation*, mutation responsiveness, toast wiring on real network errors, segment add/edit/delete persistence, and per-trip share URLs. The post-login portion of the brief (sections 3–8) is **not covered**. A follow-up session with the allowlist updated is required.

What I *did* cover with the sandbox:
- All unauthenticated routes on desktop + mobile.
- The `/auth/callback` error-rendering states.
- The "share link not found" public view.
- Mobile vs desktop parity comparison for everything reachable without a session.

## Bugs

### [SEV: blocker] Microsoft sign-in unreachable — sandbox network policy denies MS login CDNs
**Where:** `/login` and `/m/login`, both desktop and mobile. Hits the wall at `https://login.microsoftonline.com/...` once you click "Sign in with Microsoft".
**Repro:**
  1. Open `/login` (or `/m/login`) in a Chromium driven by Playwright inside the Claude Code remote execution sandbox.
  2. Click "Sign in with Microsoft".
  3. Wait — page lands on `login.microsoftonline.com/common/oauth2/v2.0/authorize?...&redirect_uri=https%3A%2F%2Facpcnnchrgoepygukxxs.supabase.co%2Fauth%2Fv1%2Fcallback&...`.
  4. The page body is completely empty.
**Expected:** Microsoft login UI renders with email field; can type `itinlytest@outlook.com`, submit, type password, and bounce back to `preview.itinly.app/auth/callback` with a Supabase session.
**Actual:** `document.body.innerText === ''`, `document.querySelectorAll('input').length === 0`, `document.querySelectorAll('form').length === 0`. The MS login client JS and CSS fail to load because the sandbox returns `403 host_not_allowed` for `aadcdn.msftauth.net` and `aadcdn.msauth.net`. The browser also logs CORS errors against those hosts (the sandbox MITM proxy strips headers).
**Notes:** This is a **sandbox limitation, not a product bug** — Microsoft's CDN is a normal public dependency of `login.microsoftonline.com` and would work in a real browser. Filing as blocker because it stops the QA pass cold. To unblock the next session, the sandbox connect-src allowlist needs:
- `aadcdn.msftauth.net`
- `aadcdn.msauth.net`
- `acctcdn.msftauth.net` (some account flows use this)
- `login.live.com`, `account.live.com`, `js.live.com` (personal-MSA flows often relay through live.com)

Screenshots: `docs/qa-screenshots/03-after-ms-click.png`, `docs/qa-screenshots/03b-ms-state.png`.

### [SEV: high] Sentry browser SDK blocked by app's own Content Security Policy
**Where:** Every page in the app — first reproduced on `/`, `/welcome`, `/login`, `/auth/callback`, both viewports.
**Repro:**
  1. Open any page on `https://preview.itinly.app` with devtools.
  2. Observe console.
**Expected:** Sentry browser SDK can POST to its configured ingest endpoint so frontend errors are captured. The whole point of shipping `NEXT_PUBLIC_SENTRY_DSN` in the bundle (per `CLAUDE.md` env-var table) is to enable browser-side error reporting.
**Actual:** Every Sentry envelope POST is refused by CSP. Example console error:
```
Connecting to 'https://o4511011332554752.ingest.us.sentry.io/api/4511326371446784/envelope/?sentry_version=7&sentry_key=7400a948b3af40616270a8dd056ed9f0&sentry_client=sentry.javascript.browser%2F10.51.0' violates the following Content Security Policy directive: "connect-src 'self' https://itinly-pre…"
Fetch API cannot load https://o4511011332554752.ingest.us.sentry.io/... Refused to connect because it violates the document's Content Security Policy.
```
Sentry is configured *and* loaded — it just can't call home. Frontend error visibility is silently zero in this build.
**Notes:** The `connect-src` directive needs `https://*.ingest.us.sentry.io` (or the project-specific host). This is independent of the sandbox; the CSP is set by the app's own Next.js config / proxy. Filed as **high** because it defeats the production error-reporting setup that `CLAUDE.md` explicitly documents.

Screenshot: any of the `04-*` PNGs has this error in the captured console log; see `04-desktop-root.png`.

### [SEV: medium] React hydration error (#418) on `/auth/callback`
**Where:** `/auth/callback` and `/auth/callback?error=...`, both desktop and mobile.
**Repro:**
  1. Visit `https://preview.itinly.app/auth/callback` directly.
  2. Open devtools console.
**Expected:** No React errors. Page either shows the error state cleanly (no `code` / `session_id` in URL) or processes the callback.
**Actual:** `pageerror` fires immediately:
```
Minified React error #418; visit https://react.dev/errors/418?args[]=HTML&args[]= for the full message or use the non-minified dev environment for full errors and additional helpful warnings.
```
React error #418 = "Hydration failed because the initial UI does not match what was rendered on the server." This implies the callback page renders different content on the server than the client — probably because it reads `window.location.hash` or `searchParams` in a way that's different SSR-vs-CSR. The user-visible page eventually settles, but the hydration error means the entire React tree is thrown away and re-rendered, which is slow and can cause flash. Worth fixing — this is the page users land on every successful sign-in.
**Notes:** Reproduces on both `/auth/callback` and `/auth/callback?error=access_denied&error_description=user_cancelled`. The visible UI is fine ("Sign-in could not be completed." / "Sign-in was cancelled." + "Back to sign-in" button) but the console error is real and stems from app code, not an external dep. Suggest auditing the callback page for server-vs-client divergence (likely `useSearchParams()` or `window.location.hash` parsing inside a Server Component path, or a `Date.now()` / `new Date()` rendered without `suppressHydrationWarning`).

Screenshot: `docs/qa-screenshots/04-desktop-auth_callback.png`.

### [SEV: medium] Mobile `/m/settings` returns 404 — no mobile mirror for `/settings`
**Where:** `/m/settings` on mobile UA.
**Repro:**
  1. Set a mobile viewport / user agent.
  2. Navigate to `https://preview.itinly.app/m/settings`.
**Expected:** A mobile settings page that mirrors the desktop `/settings` route. `CLAUDE.md` is explicit: "Every UX change must land in both the desktop site … and the mobile site at `apps/web/src/app/m/...`." `apps/web/src/app/m/` already contains `settings/` per directory listing — but the route doesn't render (returns 404).
**Actual:** `404 | This page could not be found.` — both the route and the in-app navigation to it appear broken.
**Notes:** `ls apps/web/src/app/m/` returns: `login  page.tsx  settings  shared  trip` — there *is* a `settings/` folder, so either it's missing a `page.tsx`, or the route is intentionally gated behind auth and returns 404 instead of redirecting to login. Either way, the failure mode is wrong: an authenticated mobile user has no way to reach settings, and an unauthenticated user gets a confusing 404 instead of being asked to sign in (compare desktop `/settings`, which I couldn't reach to verify but should follow the auth redirect pattern).

Screenshot: `docs/qa-screenshots/04-mobile-m_settings.png`.

### [SEV: medium] `_rsc` and `/health` background requests fail with `net::ERR_ABORTED` on every navigation
**Where:** Console / Network panel on every page transition. Example URLs:
- `GET https://preview.itinly.app/?_rsc=cbggn :: net::ERR_ABORTED`
- `GET https://preview.itinly.app/welcome?_rsc=17yrj :: net::ERR_ABORTED`
- `GET https://itinly-preview.up.railway.app/health :: net::ERR_ABORTED`
**Repro:**
  1. Open any unauthenticated page on `preview.itinly.app`.
  2. Navigate to any other page.
  3. Watch the Network panel.
**Expected:** Either no spurious requests, or requests that complete cleanly. The `_rsc` requests are Next.js React Server Component prefetches; the `/health` ping is the app's own backend health check.
**Actual:** Many `_rsc` requests abort on navigation (likely racing the next navigation — fine but noisy). The `/health` ping consistently aborts. From the sandbox I can confirm the backend itself is fine — `curl -sI https://itinly-preview.up.railway.app/health` returns `200 OK` — so the browser request is dying. Most likely CORS preflight failure: I didn't see explicit `Access-Control-Allow-Origin` errors for `/health` but the request never completes from the browser. If this is the health-poll the navbar status indicator uses, the indicator is silently broken for everyone.
**Notes:** Filing medium because the `_rsc` aborts are background noise but the `/health` failure may indicate either a CORS misconfig on the Railway backend's `/health` endpoint or a fetch that's invoked too early in the page lifecycle and torn down. Worth investigating; if the health check isn't user-visible, downgrade to low.

### [SEV: low] Inconsistent copy on the "share link not found" view between desktop and mobile
**Where:** `/shared/<unknown-id>` desktop vs `/m/shared/<unknown-id>` mobile (mobile UA redirects there automatically).
**Repro:**
  1. Open `https://preview.itinly.app/shared/nonexistent-share-id` on desktop.
  2. Open the same URL on mobile UA — observe redirect to `/m/shared/nonexistent-share-id`.
**Expected:** Identical copy and CTA. `CLAUDE.md`: "Every UX change must land in both the desktop site … and the mobile site at apps/web/src/app/m/...".
**Actual:**
| | Desktop | Mobile |
|---|---|---|
| Heading | "Trip not found" | "Shared trip" |
| Body | "This share link may have **expired or been removed.**" | "This share link may have **expired or been revoked.**" |
| CTA | "Open itinly" | "Try again" |

Different verb ("removed" vs "revoked" — they aren't synonyms; *revoked* implies the owner pulled it, *removed* implies deletion), different heading, different CTA destination/wording. Pick one and align.
**Notes:** "Revoked" is more accurate for share links (matches the brand vocabulary in `CLAUDE.md` which talks about share-link revoke flows). "Try again" CTA on mobile is also weird — there's nothing to try again; the link is gone. "Open itinly" is more useful. Recommend: heading "Shared trip" everywhere, body "This share link may have expired or been revoked." everywhere, CTA "Open itinly" everywhere.

Screenshots: `docs/qa-screenshots/04-desktop-shared_nonexistent-share-id.png`, `docs/qa-screenshots/04-mobile-shared_nonexistent-share-id.png`.

### [SEV: low] Mobile "Install itinly on your phone" banner is rendered inline on every mobile page including 404
**Where:** Every page under iOS Safari user agent, including `/m/login`, `/m/settings` (which is 404), `/m/shared/...`.
**Repro:**
  1. Use a mobile UA (e.g. iPhone Safari).
  2. Visit any page.
**Expected:** PWA install prompt is shown sparingly — typically once per session, dismissible, or only on the first/landing screen. Not stacked on top of every page including error states.
**Actual:** "Install itinly on your phone — Tap the Share icon in Safari, then Add to Home Screen." appears as inline static content at the top of every mobile page, *including* the 404 page (`/m/settings`) and the share-link-not-found page. There's no dismiss control visible in the captured screenshots.
**Notes:** This is mostly polish, but the cumulative effect across navigations is noisy — particularly on error pages where the install banner appears *above* the error message. Suggest making it dismissible, persisting the dismissal in localStorage, and not rendering it on 404/error states.

Screenshots: `docs/qa-screenshots/04-mobile-m_settings.png`, `docs/qa-screenshots/04-mobile-shared_nonexistent-share-id.png`.

## Areas explicitly NOT tested (need next session)

The following sections of the brief are entirely uncovered because they require an authenticated session that this sandbox cannot establish:

1. **Trip CRUD** — create / rename / change dates / delete on `/trips/*` and `/m/trip/*`, plus the rapid-back-to-back optimistic-update behavior `CLAUDE.md` calls out.
2. **Email scan** — connect Outlook, run a scan, review parsed segments, accept / reject / edit one, re-scan dedup.
3. **Segments** — add / edit / delete every supported type (flight, hotel, train, car, activity, breakfast/lunch/dinner, show, tour, cruise, transport). Overnight flights, multi-day hotels, missing-field cases.
4. **Todos** — per-trip todos in each category, rapid check/uncheck, reorder, delete.
5. **Sharing** — create link, open incognito, verify read-only, revoke, confirm 404.
6. **Calendar sync** — Outlook Calendar push, wall-clock TZ correctness per `CLAUDE.md`.
7. **Desktop ↔ mobile parity inside authenticated UI** — every `/trips/*` ↔ `/m/trip/*` parity check.
8. **Polish/regression checks** — keyboard nav, focus rings, dark mode, broken images, console errors inside the authenticated UI, mobile viewports at 375×812 and 414×896.

The next QA session should pick up at sign-in (using the unblocked allowlist) and work through 1–8.
