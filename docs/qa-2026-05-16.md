# QA Report — 2026-05-16 (staging shakedown)

**Tester:** Claude (Opus 4.7), running a Playwright + Chromium driven session against `https://preview.itinly.app`.
**Account:** `itinlytest@outlook.com` (Microsoft consumer / Outlook).
**Branch:** `claude/qa-test-travel-app-oGGYg`.

## Summary

14 bugs filed across desktop and mobile authenticated surfaces. No blockers — every core flow completes — but the **Scan emails** dialog has a hard "stuck on Connect" bug that blocks actually testing email-to-segment ingest, and several mobile parity gaps (no status control, no Import email / Export, random Japan map) point at the desktop being the source of truth and the mobile shell being built off it more haphazardly. The rapid-click + remount issue on todo checkboxes and the status pill violates a `CLAUDE.md` invariant directly.

### Bugs by severity

| Severity | Count |
|---|---|
| blocker | 0 |

**High bugs (6):**
- `/trips` is reachable when logged out (no redirect to `/login`).
- Scan emails dialog stays on "Connect an email account" after Outlook is successfully connected — hard block on testing email ingest.
- Mobile trip Map renders Japan when the trip has no geocoded segments.
- No trip-status control on the mobile trip-detail header (status only reachable behind ⋮ → Edit trip).
- "Set city" on a day is free-text with no autocomplete and never geocodes — Map tab can't render based on day cities.
- Share-to-edit is Gmail-only, but the dialog accepts arbitrary non-Gmail addresses with no validation → silent dead-end shares.

### What worked smoothly

- Microsoft sign-in completes end-to-end and lands on the home `/` page with trips loaded. Localstorage + Supabase JWT persist correctly.
- Trip CRUD: create, inline rename, inline date edit, status cycle (Planning → Active → Completed → Cancelled → Planning), and delete with a confirmation dialog all work.
- Segment add: full Type dropdown (Flight, Train, Car Rental, Car Service, Other Transport, Hotel, Breakfast, Brunch, Lunch, Dinner, Activity, Show, Tour, Cruise = 14 types), flight-specific fields including airline / flight # / departure & arrival airports, "More options" expansion for cost + cabin + seat + confirmation #.
- Segment renders with the correct rail/icon colours per the segment-type token system (Flight in blue) — desktop + mobile.
- Share link: create, copy URL, view in incognito (read-only — no edit buttons), revoke, and the revoked link cleanly shows "This share link may have expired or been revoked. → Open itinly".
- Calendar sync: Outlook Calendar connect, "Sync to Calendar…" dialog, success toast "1 event synced to your calendar". After sync, the menu correctly reflects "Calendar synced (1)" on both desktop and mobile.
- History tab logs each action (create todo, add segment, share + revoke, etc.) with author and timestamp.
- "Cancelled" trip status correctly uses the `--status-danger` token on the card pill (verified by visual inspection).
- Desktop trip cards group correctly into NOW / UPCOMING / PAST buckets based on today's date (`2026-05-16`). "In 33 days", "In 46 days" relative-time badges are accurate.

### Test environment notes

- Vercel deployment protection gate is satisfied with the share-link bypass URL (`?_vercel_share=…`), which sets a `_vercel_jwt` cookie. After that the rest of the app is reachable normally.

To complete sign-in for this QA session, requests to that host were stubbed locally with a meta-refresh to the `ru` redirect target. **This is a test-harness workaround only and not a product bug** — real users do not hit a sandbox allowlist.

### [SEV: medium] Long trip names (>~80 chars) overflow the trip-detail header horizontally — no `word-break`

**Where:** Trip detail desktop (`/trips?id=…`), the `<h1>` row.

**Repro:**
1. Create a trip whose name is a long single token (e.g. 200 `X`s, or a real-world long name without spaces).

### [SEV: high] "Set city" on a day is a free-text input with no autocomplete and never geocodes — the Map tab can't render based on day cities

**Where:** Trip detail desktop, day row "Set city" → inline edit.

**Repro:**
1. Open a trip with at least one day.
2. Click "Set city" on the day. An inline input appears (no label, no placeholder).
3. Type a real city name (e.g. "Tokyo").

**Notes:** Two clean fixes — either autocomplete + geocode (preferred), or just clarify the field's purpose with a placeholder. **Where:** Trip detail → Share → "Can edit · Specific Gmail" tab.

**Repro:**

### [SEV: medium] Rapid clicks on the todo checkbox detach the button from the DOM, dropping clicks

**Where:** Trip detail desktop, right-hand To-do panel. The "Mark complete" / "Mark incomplete" toggle button.

**Repro:**
1. Open a trip with at least one todo.
2. Programmatically (or by hammering the mouse) click the todo's checkbox button 6 times in <100 ms.
3. Wait for the network to settle.

**Expected:** Per `CLAUDE.md`: "Check that toggling the same checkbox five times in a row works without the UI freezing." Optimistic updates should let each click flip the visible state. 6 clicks → ends back where it started.

**Actual:** Playwright reports `element was detached from the DOM, retrying` between successive clicks — meaning the optimistic re-render is swapping the `<button>` element rather than just updating its attributes. Real users hitting this don't get an error; they just get clicks silently swallowed because the click event lands on a transient (gone) DOM node.

**Notes:** Same root cause as the status-pill rapid-click bug. The fix is probably to update the SVG/`aria-label` in place instead of remounting the button (stable identity React-side, plus avoid re-rendering ancestors that wipe the child). Screenshot: `docs/qa-screenshots/230-before-rapid.png` (single todo).
