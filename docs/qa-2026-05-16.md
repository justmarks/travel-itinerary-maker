### [SEV: high] Shrinking a trip's date range silently destroys segments outside the new range — no warning, no history entry, no way to recover

**Where:** Trip detail desktop. Inline date editor on the trip header.

**Repro:**
1. Open a trip that has segments on multiple days (e.g. a hotel on day 1 and a flight on day 5).
2. Click the trip date range. Inline date inputs appear with checkmark / X.
3. Shrink the end date so the flight's date falls outside the new range (e.g. trip Oct 1-Jan 9, shrink end to Oct 3, flight is on Oct 5).
4. Click the checkmark to save.
5. After save, extend the end date back to Oct 10.

**Expected:** Either (a) a confirmation dialog before the save: "Shrinking the trip will remove 1 segment (UA 850 SFO → NRT on Oct 5). Are you sure?" with explicit Confirm / Cancel; or (b) the segments are preserved (just hidden) and reappear when the date range is extended back to cover them.

**Actual:**
1. The save goes through with **no warning**. The flight on Oct 5 silently disappears from the UI.
2. The History tab logs only `dates 2026-10-01–2027-01-09 → 2026-10-01–2026-10-03` — **no entry for the segment that was lost**.
3. Re-extending the end date back to Oct 10 does **not** bring the flight back. The segment is gone.

**Notes:** This is a real data-loss path. A user moving their trip's end date in one step loses any segments that fall outside the new range, with no warning and no audit trail. Two clean fixes: warn before save (preferred — the user may want to keep the segment and just adjust dates later) or soft-delete with a "Restore" option in History. Currently neither. Screenshots: `docs/qa-screenshots/804-after-shrink-oct3.png` (after shrink — flight gone), `docs/qa-screenshots/810-extended.png` (after extend — flight still gone), `docs/qa-screenshots/815-history.png` (history with no segment-deletion entry).

### [SEV: high] React-Query cache (`itinly-rq-cache-v1`) persists in `localStorage` after Sign out — next user / observer can read it

**Where:** User menu → Sign out. Desktop.

**Repro:**
1. Sign in.
2. Open + use the app so React Query has cached trips, share rules, etc.
3. Click your avatar → "Sign out". You're redirected to `/login` (correct).
4. Open dev-tools → Application → Local Storage → `https://preview.itinly.app`.

**Expected:** All app data (`itinly-rq-cache-v1`, `itinly-auth`, `sb-acpcnnchrgoepygukxxs-auth-token`, etc.) is purged on sign-out.

**Actual:** `itinly-auth` and the Supabase token *are* cleared (good). But **`itinly-rq-cache-v1` is left intact** — it holds the previous user's serialized trip data (titles, dates, segments, todos, history). Anyone else who uses the same browser before another sign-in can read it via dev-tools, and the next sign-in will hydrate from this cache on first paint before the new user's data overwrites it (potentially flashing the previous user's trips for a frame).

**Notes:** Add `localStorage.removeItem('itinly-rq-cache-v1')` (and any other React-Query persistence keys) to the sign-out handler. Belt-and-braces also clear the cache on sign-in if it doesn't match the current `userId`. Also keep `itinly:hint:intro-tour` — that's a per-device UI flag and is fine to persist. Screenshot: console log in `/tmp/qa/40-signout.js` output captured `LS after signout: [ 'itinly:hint:intro-tour', 'itinly-rq-cache-v1' ]`.

### [SEV: medium] Rapid clicks on the trip status pill drop most of the clicks instead of cycling through states

**Where:** Trip detail desktop, status pill next to the trip name (`/trips?id=…`).

**Repro:**
1. Open a trip detail page.
2. Programmatically (or by hammering the mouse) click the status pill 8 times in <100 ms at the same screen position.
3. Wait for the network to settle and observe the final state.

**Expected:** Per `CLAUDE.md`'s "tappable in rapid succession" rule, optimistic updates should advance one step per click. 8 clicks against a 4-step cycle (Planning → Active → Completed → Cancelled → Planning) should land 2 cycles ahead of where it started.
