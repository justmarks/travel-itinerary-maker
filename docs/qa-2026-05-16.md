## Test environment notes

- Vercel deployment protection gate is satisfied with the share-link bypass URL (`?_vercel_share=…`), which sets a `_vercel_jwt` cookie. After that the rest of the app is reachable normally.
- The Microsoft consumer-account OAuth flow goes through `ms-sso.copilot.microsoft.com`, which is not in this sandbox's egress allowlist. To complete sign-in for this QA session, requests to that host were stubbed locally with a meta-refresh to the `ru` redirect target. **This is a test-harness workaround only and not a product bug** — real users do not hit a sandbox allowlist.
- Sandbox MITM-proxy artifacts (SSL-cert errors on service-worker registration, CORS errors against `en.wikipedia.org`, `*.googleapis.com`, `ms-sso.copilot.*`, etc.) are filtered out of the bug list per the rules.

## Bugs

### [SEV: medium] `/trips` (no `id` param) shows a red "No trip selected." error instead of the trip list

**Where:** `/trips` desktop, both authenticated and unauthenticated.

**Repro:**
1. Sign in.
2. Navigate directly to `https://preview.itinly.app/trips` (no query string).

**Expected:** Either the trip list (since `My Trips` lives at `/`), or a clean redirect to `/`.

**Actual:** A page is rendered with a small `← Back` link and the text **"No trip selected."** styled in error red (`text-destructive` or similar). It looks like a hard error.

**Notes:** Same page renders the same red error even when logged out — see the next bug. The page reaches the layout of the trip-detail view but with no trip selected. Either the route should not exist, or it should fall through to the trip list. Screenshot: `docs/qa-screenshots/state-_trips.png`.

### [SEV: high] `/trips` is reachable when logged out — no redirect to `/login`

**Where:** `/trips` desktop, unauthenticated.

**Repro:**
1. Open a fresh browser context (no cookies / no Supabase token).
2. Visit `https://preview.itinly.app/trips` directly (after satisfying the Vercel share-link bypass).

**Expected:** The route is auth-gated and redirects to `/login` (the same way `/` does).

**Actual:** The page renders the trip-detail shell with `← Back` and the red "No trip selected." text. No redirect happens. (Auth-gated content isn't leaked, but the route's UX implies the user is signed in.)

**Notes:** Other authenticated entry points (`/`) correctly redirect to `/login`. This route appears to be an outlier. Screenshot: `docs/qa-screenshots/bug01-trips-unauth-no-redirect.png`.

### [SEV: medium] Rapid clicks on the trip status pill drop most of the clicks instead of cycling through states

**Where:** Trip detail desktop, status pill next to the trip name (`/trips?id=…`).

**Repro:**
1. Open a trip detail page.
2. Programmatically (or by hammering the mouse) click the status pill 8 times in <100 ms at the same screen position.
3. Wait for the network to settle and observe the final state.

**Expected:** Per `CLAUDE.md`'s "tappable in rapid succession" rule, optimistic updates should advance one step per click. 8 clicks against a 4-step cycle (Planning → Active → Completed → Cancelled → Planning) should land 2 cycles ahead of where it started.

**Actual:** Starting from "Completed", 8 rapid clicks advanced the pill exactly **one** step to "Cancelled" — the other 7 click events were dropped on the floor. (When you click + wait ~700 ms each time, the cycle works correctly; only rapid-fire breaks it.)

**Notes:** Likely the pill is remounting / repositioning between updates so the second click lands on a different element (or hits a transient gap while the optimistic UI swaps the label). CLAUDE.md explicitly calls out: "Check that toggling the same checkbox five times in a row works without the UI freezing." This is the same class of issue for a different control. Screenshot: `docs/qa-screenshots/90-rapid-mouse.png`.

### [SEV: medium] `/m/settings` returns the global 404 — mobile parity gap

**Where:** Mobile `/m/settings`.

**Repro:**
1. Sign in.
2. Visit `https://preview.itinly.app/m/settings` (or — once a Settings link exists — tap it from the mobile shell).

**Expected:** Either a mobile Settings screen that mirrors the desktop `/settings` page, or, if Settings isn't ready for mobile yet, a graceful "Open on desktop" stub rather than a 404.

**Actual:** Next.js' bare "404 — This page could not be found." default error page renders.

**Notes:** Previously flagged in the prior session — still reproducible. The desktop `/settings` route exists and is reachable from the user-menu. The mobile shell has no equivalent, and the path that would obviously exist (`/m/settings`) is unrouted. Screenshot: `docs/qa-screenshots/mob-_m_settings.png`.

### [SEV: high] Mobile trip Map renders a random Japan view when the trip has no geocoded segments

**Where:** Mobile trip detail (`/m/trip?id=…`).

**Repro:**
1. Sign in.
2. Open a trip on mobile that has segments without `lat/lng` (e.g. a flight whose departure/arrival hasn't been resolved).
3. Look at the inline mini-map above the day pills.

**Expected:** Match the desktop Map tab, which says **"No mappable locations yet. Add hotels, restaurants, or activities to see them here."**

**Actual:** A Google Map of Japan (centered around Tokyo / Osaka) renders. The trip has no Japan content — the only segment is a US domestic flight `AA 100 NYC → LAX`. The map is misleading.

**Notes:** Desktop/mobile parity contract (`CLAUDE.md` → "Every UX change must land in both… don't ship an affordance to one and forget the other"). Either: (a) suppress the map block when there are no resolved coordinates, or (b) render the same empty state as desktop. Probably (a) since mobile keeps the map inline rather than on a separate tab. Screenshot: `docs/qa-screenshots/130-mobile-trip.png`. Compare with desktop empty state: `docs/qa-screenshots/16-tab-map.png`.

### [SEV: low] Mobile `/m/trip` (no `id` param) shows a separate "No trip" empty state — desktop `/trips` and mobile `/m/trip` use different copy

**Where:** Mobile `/m/trip` vs desktop `/trips`.

**Repro:**
1. Visit `https://preview.itinly.app/m/trip` and `https://preview.itinly.app/trips` in the same authenticated session.

**Expected:** Both routes either redirect or render a parity empty state. The mobile copy ("No trip" header + grey centered "No trip selected.") is actually nicer than desktop's red-styled error — match desktop to mobile, not the other way around.

**Actual:** Desktop renders error-red text. Mobile renders a tidy grey empty state with a labeled header.

**Notes:** Same root issue as the first two bugs — these "no-id" routes should probably not be reachable, or should redirect. Screenshots: `docs/qa-screenshots/mob-_m_trip.png` and `docs/qa-screenshots/state-_trips.png`.
